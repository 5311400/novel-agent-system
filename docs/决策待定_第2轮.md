## ✅ 已共识（可直接写进 Skill 文件）

### Q1：错误分级阈值
**方案：三级量化 + 参考锚点，不接受纯自由裁量。**

| 级别 | 判定锚点 | 处理 |
|------|---------|------|
| **严重** | 战力越≥2大境界无代价、核心OOC无解、设定硬伤、逻辑死结、已死角色复活 | 阻断流转，打回重写 |
| **中等** | 局部OOC可圆、节奏失衡、300字内≥3次修饰词滥用、轻度战力通胀 | 放行但弹窗建议修改 |
| **轻微** | 用词重复、散发性AI腔、标点小失误 | 仅后台备案 |

---

### Q2：自我进化机制
**方案：生成补丁报告 → 人工确认 → 总控代为落地。**

流程（10章一轮）：
1. PolishEditor 汇总错误数据，输出标准化补丁报告（高频错误类型 + 对应规则缺陷 + 修改草案）
2. 报告推送至用户，由人工审核
3. 确认通过后，Coordinator 重载更新全 Agent Skill
4. 未确认的补丁不生效

**禁止 Agent 自动修改 Skill 配置。**

---

### Q3：文风定位
**方案：爽文快节奏为主 + 热血战斗穿插 + 古风点缀 ≤15%。**

- 常态（~70%）：短句、对话密、每章一个爽点爆 + 钩子结尾
- 战斗（~20%）：切换到热血战斗，动作短促、冲击力词汇密集
- 情感/环境（≤10%）：古风点缀，但单段环境描写不超过 200 字

---

### Q4：AI腔检测严格度
**方案：宽松模式，500字滑动窗口累计≥3次才标记。战斗场景放宽到5次。**

- 单次 AI 腔不单独上报（减少噪音）
- 达标后统一批量标注 + 附带修改建议
- 豁免项：人物内心独白、古风书面化修辞不判定为 AI 腔

---

### Q5：状态机切换条件
**方案：都返回 + 快照版本校验通过 → 自动流转，人工不介入。**

三重锁：
1. **产出就绪**：前置 Agent 全部返回 `status: success`
2. **版本校验**：快照版本对比 → 过期驳回重跑
3. **关联校验**：跨档案引用无断裂

---

### Q8：节拍表 → 正文松紧度
**方案：柔性展开 + 首尾刚性锚点 + 字数护栏。**

- 节拍是骨架方向，正文在约束内可即兴发挥
- **刚性约束**：不删节拍、不倒序、首尾锚点必须对齐（开头对上章悬念，结尾不漏高潮）
- **字数护栏**：节拍预估字数 ±30%，超限触发压缩/扩写指令

### Q6：子Agent重度失败怎么走

**场景**：重试1-2次仍失败、返回乱码/空数据。

**选项A（5/6家支持）→ 总控降级保底写**
- Coordinator 用保守 Prompt 强行推完本章
- 产出具元数据 `[fallback: true]` / `[需人工精修]`
- 推送告警提醒用户回头改
- **理由**：不阻断流水线，保进度

**选项B（豆包坚持）→ 直接暂停等用户**
- 不降级总控代写，暂停流程推送告警
- **理由**：总控代写可能导致产出一致性崩盘

**你的选择：**
> 选项B直接暂停等用户

## ⚠️ 待你选择（第2处）

### Q7：跨档案关联校验——独立文件 vs 内嵌WorldBuilder

**选项A（豆包/Kimi/ChatGPT/智谱清言 4家支持）→ 独立 `consistency-rules.md` 全局共享 Skill**
- 所有 Agent 统一加载调用，一处修改全链路生效
- 校验规则：遍历所有档案 ID，检测引用是否存在、无删除失效条目

**选项B（千问/DeepSeek 2家支持）→ 内嵌在 WorldBuilder 的 Skill 中**
- 跨文件引用是世界观构建的专属职责
- 其他 Agent 不需要关心底层 ID，只需确认"地名存不存在"

**ChatGPT 提出的混合方案（参考）**：
- 写 `consistency-rules.md` 作为**共享校验库**（所有 Agent 读取时做被动校验）
- WorldBuilder 的 Skill 中额外加**主动校验义务**（写入时扫描全局防引用断裂）
- 双层设计：写时防断裂 + 读时防遗漏

**你的选择：**
> ChatGPT 提出的混合方案

## P1：DeepSeek 实施歧义（已拍板）

### 问题A：重度失败暂停后的唤醒机制
**决策：超时自动保存 + 提醒，不无限期挂起。**

- 总控暂停当前章节流程，生成 halt-report.md
- 推送告警给用户
- 若 **2 小时内用户未响应** → 自动将当前草稿保存至 runtime/interrupted/ 目录
- 发送浓缩摘要提醒给用户（含：哪个Agent失败了、失败类型、上下文快照位置）
- 用户回来后可在中断点恢复
- 超过 24 小时仍未响应 → 将中断信息归档至 data/changelog.md，标记为 `[暂停-超时归档]`

### 问题B：战力越级与战斗场景的交叠
**决策：200字内交代了合理性因素 → 降为中等；否则仍严重。**

判定流程：
1. PolishEditor 检测到"战力越≥2大境界"
2. 检查该段是否为战斗场景（调用 StyleGuard 的 scene_type 判断）
3. 若是战斗场景 → 检查触发点前 200 字内是否交代了外力因素（道具/地利/合体技/封印解除等）
4. 若已交代 → 降为中等（建议补铺垫，不阻断）
5. 若未交代或解释牵强 → 仍按严重处理（阻断流转）

已写入 polish-review.md 的严重级一节。

## P2：ChatGPT 工程缺口

### 缺口1：错误类型分类（Retry Policy）

当前只有"重试2次→暂停"，但没有区分哪些错误可重试、哪些必须直接挂。

| 错误类型 | 可重试？ | 处理 |
|---------|---------|------|
| API Timeout | 可重试 | 自动重试2次，间隔10秒 |
| Rate Limit | 可重试 | 自动重试2次，间隔10秒 |
| JSON Parse 错误 | 可重试 | 自动重试2次，间隔10秒 |
| Network 波动 | 可重试 | 自动重试2次，间隔10秒 |
| Token Overflow | 可重试 | 自动重试（加大上下文窗口） |
| **Skill 缺失** | **不可重试** | 直接 Fail，生成 halt-report |
| **Prompt 错误** | **不可重试** | 直接 Fail，生成 halt-report |
| **Schema 错误** | **不可重试** | 直接 Fail，生成 halt-report |
| **上下文损坏** | **不可重试** | 直接 Fail，生成 halt-report |
| **版本冲突** | **不可重试** | 直接 Fail，回退到版本校验环节 |

**写入位置**：shared/02_data-access-rules.md 或 runtime/.error-codex 规范文件。

### 缺口2：锁的 TTL

已有原子锁协议，但缺具体超时和心跳细节：

```
.write.lock TTL：
  acquire_ts → 锁创建时间
  expire_ts = acquire_ts + 120s（默认过期）
  heartbeat：写入流程内部每 30s 刷新 expire_ts
  超过 expire_ts → 视为僵尸锁，自动回收
  回收记录到 runtime/.violations.log

.read.shared TTL：
  只读标记每 60s 检查一次
  若持有者超过 120s 无活动 → 自动销毁标记
```

已写入 coordinator.md 的原子锁协议一节和 data/.lock-policy.md。

### 缺口3：快照生命周期

当前 GV 版本号一直在递增，但未定义旧快照的保留策略：

```
快照保留策略：
  最近 20 章：保留完整快照（含人物/世界观/章节正文）
  >20 章且 ≤100 章：仅保留 Diff（与上一版本的差异）
  >100 章：仅保留元数据（GV、时间戳、章节摘要链接）
  
  存档位置：
    完整快照：runtime/snapshots/full/
    Diff 快照：runtime/snapshots/diff/
    元数据：data/INDEX.md（GV 索引）
    
  维护时机：
    每发布一章终稿后自动触发快照
    每 10 章执行一次旧快照归档
```

### 缺口4：Lore Manager（档案管理员）Agent 提案

ChatGPT 建议新增一个 Agent，与 WorldBuilder 职责分离：

| 维度 | WorldBuilder（创造者） | LoreManager（存储者） |
|------|----------------------|---------------------|
| 职责 | 构思/填充世界观内容 | CRUD 操作、版本管理 |
| 创造内容 | ✅ 是 | ❌ 否 |
| 维护引用 | ❌ 不负责（写时校验即可） | ✅ 统一维护 |
| 快照管理 | ❌ 不负责 | ✅ 负责 |
| 孤儿清理 | ❌ 不负责 | ✅ 每10章扫描 |
| Diff 生成 | ❌ 不负责 | ✅ 负责 |

**当前决策**：暂不新增独立 Agent。将 LoreManager 的职责拆入两个已存在的 Skill：
- consistency-rules.md（共享规则）承担被动校验 + 孤儿检测
- WorldBuilder Skill 承担主动校验（写入时扫描引用）
- Coordinator Skill 承担快照管理（补丁落地流程中已有版本记录）

若后期发现 WorldBuilder 负担过重 → 考虑拆分出独立 LoreManager。

## P3：智谱清言潜在风险点（需在实施阶段验证）

### 风险1：Q1「越2级」需要机器可读的战力等级表
当前 PolishEditor 检测"越≥2大境界"时，依赖它自己从 power-system.md 读取境界排序。
- 风险：power-system.md 是自然语言文件，Agent 解析可能出错
- 缓解：在 data/world/power-system.md 中加入机读字段 `cultivate_rank_order: [炼气,筑基,金丹,元婴,化神]`，PolishEditor 直接按数组索引判定
- 验证时机：第一次实装时测试 polish-review.md 能否正确解析

### 风险2：Q4 滑动窗口在多 Agent 异步场景下的一致性
AI腔检测使用 500 字滑动窗口——若章节正文由多段异步产出，窗口可能跨段失效。
- 风险：ChapterWriter 写完 → PolishEditor 质检时，窗口按整章连续文本计算，本应无问题。但若引入分片写作（不同Agent写不同段落），窗口会断裂
- 缓解：目前 ChapterWriter 是串行写整章，窗口不跨 Agent。若未来改为分片 → 滑动窗口在 PolishEditor 统一计算，不在写作阶段算
- 验证时机：维持现方案即可，分片时再上线滑动窗口聚合

### 风险3：Q8 ±30% 字数护栏依赖节拍表预估质量
如果节拍表本身的预估字数就偏离实际（比如预估 500 字但实际需要 1000 字才能讲清楚），正文要么被强制压到 650 字导致信息缺失，要么节拍表被废除。
- 风险：节拍表质量差 → 护栏形同虚设或反成负担
- 缓解：
  1. 初期护栏放宽松（±50%），跑 10 章收集数据后再收紧到 ±30%
  2. ChapterWriter 产出节拍表时自动登记"预估置信度"（high/medium/low），low 置信度的节拍暂不加护栏
- 验证时机：第 5 章后分析节拍表 vs 正文的字数偏差分布
